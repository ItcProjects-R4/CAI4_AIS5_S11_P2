# Data Sources

This page describes the two raw data sources that feed the ETL pipeline -- the CRM CSV export and the Excel customer spreadsheet. Understanding the source schemas and known quality issues is essential before working on transformations.

---

## Overview

| Source | Format | Owner | Update Frequency | Location |
|---|---|---|---|---|
| CRM Export | CSV (UTF-8) | CRM System | Weekly | `data/raw/crm_customers_YYYYMMDD.csv` |
| Excel Spreadsheet | Excel (.xlsx) | Sales Team | Weekly | `data/raw/excel_customers_YYYYMMDD.xlsx` |

Both files are placed in `data/raw/` manually before each pipeline run. The pipeline reads them, merges them, and produces a single clean output.

> **Rule:** Never edit files in `data/raw/`. They are the original source of truth. If there is an error in a raw file, fix it at the source system and re-export.

## Source 1 -- CRM Export (CSV)

### What It Is

The CRM system exports a full snapshot of all customer records as a CSV file. This file is generated by the CRM team and placed in `data/raw/`.

### File Naming Convention

```
crm_customers_YYYYMMDD.csv
```

Example: `crm_customers_20240601.csv`

### File Format

- Encoding: UTF-8
- Delimiter: comma (`,`)
- Header: yes (first row contains column names)
- Quote character: double quote (`"`) used around fields that contain commas
- Line ending: CRLF (Windows) or LF (Unix)

### Schema

| Column | Data Type | Example | Description |
|---|---|---|---|
| `customer_id` | Integer | `1001` | Unique customer identifier -- primary key |
| `full_name` | String | `John Doe` | Customer full name |
| `email` | String | `john@example.com` | Contact email address |
| `phone` | String | `01012345678` | Phone number in local format |
| `signup_date` | String (date) | `2023-05-10` | Date customer registered, format `YYYY-MM-DD` |
| `country` | String | `Egypt` | Country of residence (full name) |
| `segment` | String | `Premium` | Customer tier: `Premium`, `Standard`, or `Basic` |

### Known Quality Issues

| Issue | Impact | How Pipeline Handles It |
|---|---|---|
| `phone` uses inconsistent formats (spaces, dashes, no country code) | Hard to compare or display | Normalised by Derived Column transformation |
| `signup_date` is blank for records imported before 2022 | NULLs in output | Allowed to be NULL -- documented exception |
| `segment` values use mixed casing (`premium`, `PREMIUM`, `Premium`) | Inconsistent grouping in reports | Converted to uppercase by transformation |
| Duplicate `customer_id` can occur in incremental exports | Inflated row counts | Deduplicated by Aggregate step |
| Some rows have extra spaces in `full_name` | Dirty name data | Trimmed and title-cased by transformation |

## Source 2 -- Excel Spreadsheet

### What It Is

The sales team maintains customer records in an Excel workbook updated weekly. The file is exported from their system and placed in `data/raw/`.

### File Naming Convention

```
excel_customers_YYYYMMDD.xlsx
```

Example: `excel_customers_20240601.xlsx`

### File Format

- Format: Microsoft Excel Open XML (.xlsx)
- Relevant sheet: `Customers` (always the first sheet)
- Header: yes (row 1 contains column names)
- Date storage: Excel may store dates as serial numbers (e.g. `45048`) or as formatted strings (`10/05/2023`)

### Schema

| Column | Data Type | Example | Description |
|---|---|---|---|
| `CustomerID` | Integer | `1001` | Unique customer identifier -- primary key |
| `Name` | String | `John Doe` | Customer full name |
| `EmailAddress` | String | `John@Example.COM` | Contact email (often mixed case) |
| `PhoneNumber` | String | `+20-101-2345678` | Phone with country code prefix |
| `JoinDate` | Date / String | `10/05/2023` | Date customer joined, format `DD/MM/YYYY` |
| `Country` | String | `EGY` | Country code (ISO 3166-1 alpha-3) |
| `CustomerSegment` | String | `PREMIUM` | Customer tier (uppercase) |

### Known Quality Issues

| Issue | Impact | How Pipeline Handles It |
|---|---|---|
| `JoinDate` stored as Excel serial number | Date is unreadable as a string | ADF parses using `toDate()` with Excel epoch conversion |
| `Country` uses 3-letter ISO codes instead of full names | Mismatches CRM country values | Mapped to full country name via lookup in Derived Column step |
| `EmailAddress` is mixed case | Inconsistent -- same email may appear as different values | Lowercased by transformation |
| Merged cells in Excel produce blank rows in ADF | Unexpected NULLs | Caught by the Filter step (null CustomerID) |
| `CustomerID` may overlap with CRM IDs | Same customer in both sources | Resolved by Aggregate deduplication step |

## Overlap Between Sources

Because both systems record customer information independently, the same customer may appear in both the CRM export and the Excel file with slightly different field values.

### Resolution Strategy

The pipeline prefers the CRM record as the primary source of truth:

1. Both sources are cleaned independently and then **unioned** together.
2. The **Aggregate** step deduplicates on `CustomerID`, preferring the CRM version of each field.
3. The `SourceSystem` column (`CRM` or `Excel`) records which source a field value came from.

## Adding New Source Files

When a new version of a source file is ready:

1. Name the file following the convention (see above).
2. Place it in `data/raw/`.
3. If the schema has changed (new column, renamed column), update the ADF dataset (`ds_crm_source` or `ds_excel_source`) and re-export the JSON.
4. Run the pipeline in **Debug** mode first to verify the new file is read correctly.
5. Commit the new file and any updated ADF JSON.

## Source File Checklist (Before Each Run)

Use this checklist before triggering a new pipeline run:

- [ ] New CRM file is in `data/raw/` with the correct name and date
- [ ] New Excel file is in `data/raw/` with the correct name and date
- [ ] Both files have a header row with the expected column names
- [ ] No columns have been added, removed, or renamed since the last run
- [ ] File encoding is UTF-8 for the CSV
- [ ] Excel file sheet name is still `Customers`
